프론트엔드 아키텍처 설계

여행 일정 공유 서비스 'Michiki'

① View (화면)

UI & 컴포넌트

유저 인터페이스를 구성하는 재사용 가능한 컴포넌트 집합

  • 주요 페이지 UI:
    • 메인, 대쉬보드, 스크랩 페이지
  • 여행 일정 상세 페이지 컴포넌트:
    • GoogleMap: 지도 표시 영역
    • Autocomplete: 장소 검색 자동완성
    • DatePicker: 날짜 선택 달력
    • DraggablePin: 좌측 사이드바의 드래그 가능한 핀
    • PinModal: 핀 상세정보/메모용 모달
    • CustomInfoWindow: 지도 위 장소 정보창
    • CategoryButtons: 주변 장소 탐색 버튼
  • UI 라이브러리/훅:
    • DndContext: 드래그앤드롭 환경 제공
→
↓

② Controller (로직)

이벤트 처리 & 함수

유저 이벤트에 반응하여 상태 변경을 요청
또는 API를 호출하는 함수들의 집합

  • 이벤트 핸들러 함수:
    • onLoadMap(map): 지도 초기화
    • handleAddPin(): 핀 추가
    • handleDeletePin(id): 핀 삭제
    • handlePinClick(pin): 핀 클릭 (모달 열기)
    • handleDragEnd(event): 핀 순서 변경
    • handleNearbySearch(type): 주변 장소 검색
  • API 요청 및 라우팅:
    • Google Auth, Places, Geocoding API
    • 백엔드 API (일정/핀/메모 CRUD)
    • useNavigate, useLocation (페이지 이동)
→
↓

③ Model (데이터)

전역 상태 관리

애플리케이션의 모든 데이터를 저장하고 상태 변화를 관리

  • 핵심 데이터 구조:
    • pinsByDay: 날짜별 핀(장소) 2차원 배열
    • User: 유저 정보 객체
  • UI 상태 (State):
    • dateRange: 선택된 여행 날짜 범위
    • selectedDayIdx: 현재 보고 있는 날짜 인덱스
    • infoWindow: 지도 정보창 데이터
    • searchResult: 검색 결과 (핀 후보)
    • selectedPin: 상세 보기 중인 핀 데이터
    • modalOpen: 핀 상세 모달 열림 여부
    • activeCategory: 선택된 주변탐색 카테고리

🔄 데이터 흐름 요약 🔄

[시나리오 1: 지도에 핀 추가하기]

1. View: 유저가 지도 위에서 우클릭
2. Controller: onLoadMap에서 등록한 리스너가 좌표를 감지, handleAddPin 함수를 호출
3. Model: Controller는 Model의 pinsByDay[selectedDayIdx] 배열에 새로운 핀 객체를 추가하도록 요청
4. View: pinsByDay 배열의 변경을 감지하고, 지도와 사이드바에 새로운 핀을 렌더링

[시나리오 2: 핀 순서 바꾸기]

1. View: 유저가 사이드바에서 DraggablePin을 드래그
2. Controller: DndContext가 드래그 종료를 감지 handleDragEnd 함수를 호출
3. Model: Controller는 이벤트 정보를 바탕으로 pinsByDay 배열의 요소 순서를 변경하도록 요청
4. View: 변경된 pinsByDay 배열 순서에 맞춰 사이드바의 핀 목록이 다시 렌더링

🛠️ 백엔드 아키텍처 설계

① API Layer (엔드포인트)

요청 처리 / 라우팅

클라이언트의 요청을 받고 적절한 서비스 계층으로 전달하는 역할

  • 주요 API 엔드포인트:
    • GET /plans/{id}: 여행 일정 상세 조회
    • POST /plans: 여행 일정 생성
    • PUT /plans/{id}: 일정 수정
    • DELETE /plans/{id}: 일정 삭제
  • 요청 인증 및 라우팅:
    • Spring Security를 통한 인증 처리
    • @RestController, @RequestMapping
→
↓

② Service Layer (비즈니스 로직)

비즈니스 로직 처리

실제 기능 구현과 로직 처리, 트랜잭션 관리 등을 수행

  • 핵심 서비스 로직:
    • PlanService: 일정 관련 로직 처리
    • PinService: 핀 관련 CRUD 로직
    • MemoService: 메모 저장/삭제/조회
  • 트랜잭션 및 유효성:
    • @Transactional: 트랜잭션 처리
    • DTO → Entity 변환 및 유효성 검사

↓

③ Repository Layer (데이터 접근)

DB 통신 / CRUD

데이터베이스와 직접 통신하여 데이터를 저장하고 불러옴

  • 데이터베이스 엔티티:
    • PlanEntity, PinEntity, MemoEntity
  • JPA Repository:
    • PlanRepository, PinRepository, MemoRepository
    • JPQL / Query Method 활용

🧩 인터페이스 설계 (UI Flow)

단계별 유저 흐름 제공
 시작 페이지 → 대시보드 → 상세 계획 페이지로 이어지는 3단계 UI 흐름
유저가 자연스럽게 기능을 탐색하고 이용할 수 있도록 설계


  • 소셜 로그인
     프론트엔드에서 구글의 OAuth를 통해 인증 진행 → 인증 성공시 인증 코드 전달 → JWT(JSON Web Token) 발급

  • 다국어 지원
     유저가 어플리케이션의 언어를 선택할 수 있도록 하여 선호하는 언어로 서비스 컨텐츠 제공


  • 여행 계획 CRUD 연동
     대시보드에서 보여지는 여행 계획 목록, 계획 생성 및 삭제는
    각각 GET /plans, POST /plans, DELETE /plans/{id} API와 연결


  • 로그인 여부에 따른 기능 제한
     공유 링크를 통해 접속한 로그인하지 않은 유저는 작업 참여 제한

  • 실시간 상태 동기화
     드래그 앤 드롭 후 순서 변경, 핀 추가/삭제, 색상 변경 등은 프론트에서 상태가 즉시 반영
    동시에 백엔드 API를 통해 DB에 반영되어 다른 유저와 동기화
    만약 api 요청에 실패하게 된다면, rollback

  • 실시간 협업 지원
     웹소켓 기술을 활용하여 현재 접속 중인 유저들의 목록과 각 유저의 마우스 커서 위치를 실시간으로 공유

  • 여행지 추천 연동
     추천 여행지 목록은 유저 기반 데이터를 활용하여 JSON 형태로 프론트에 전달되어 활용

  • 여행지 추천 → 일정 생성으로 연결
     추천 여행지를 클릭하면 지역 정보와 함께 일정 생성이 가능


  • 핀(장소) 데이터 처리
     계획 상세 페이지에서 장소에 대한 메모의 변경 사항은 백엔드로 전송되어 장소 수정 API를 통해 갱신

    • 🗂️ 인터페이스 흐름

    🗂️ Entity-Relationship Diagram (ERD)

    - MEMBERS와 PLANS는 다대다(N:M) 관계며 MEMBERS_PLANS_PIVOT 테이블로 연결
    - PLANS는 여러 개의 PLACES를 가질 수 있는 일대다(1:N) 관계
    - MEMBERS는 여러 개의 PLACES를 생성할 수 있는 일대다(1:N) 관계

    🗂️ 클래스 다이어그램

    👪 Table: MEMBERS

    유저 정보를 관리하는 테이블

    Column Data Type Constraints Description
    MEMBER_ID NUMBER(19) PK, NOT NULL 유저 고유 ID
    SOCIAL_ID VARCHAR2(255) NOT NULL, UQ 소셜 로그인 ID
    SOCIAL_TYPE VARCHAR2(20) NOT NULL, CHECK 소셜 로그인 타입 (GOOGLE, KAKAO...)
    NICKNAME VARCHAR2(64) NOT NULL 닉네임
    EMAIL VARCHAR2(255) NOT NULL, UQ 이메일
    PROFILE_IMAGE VARCHAR2(255) 프로필 이미지 URL
    CREATED_AT TIMESTAMP NOT NULL 생성일

    🗓️ Table: PLANS

    여행 계획의 기본 정보를 관리하는 테이블

    Column Data Type Constraints Description
    PLAN_ID NUMBER(19) PK, NOT NULL 계획 고유 ID
    TITLE VARCHAR2(255) NOT NULL 계획 제목
    START_DATE DATE 여행 시작일
    END_DATE DATE 여행 종료일
    HOST_ID NUMBER(19) FK, NOT NULL 계획 생성자(Host) ID
    SHARE_URI VARCHAR2(255) NOT NULL 공유용 고유 URI

    ✍️ Table: MEMBERS_PLANS_PIVOT

    유저와 계획의 N:M 관계를 정의하는 테이블

    Column Data Type Constraints Description
    MEMBER_ID NUMBER(19) PK, FK, NOT NULL 유저 ID
    PLAN_ID NUMBER(19) PK, FK, NOT NULL 계획 ID
    COLOR VARCHAR2(16) NOT NULL 지도에 표시될 유저별 색상
    JOINED_AT TIMESTAMP NOT NULL 참여일

    🏖️ Table: PLACES

    계획에 포함된 개별 장소를 관리하는 테이블

    Column Data Type Constraints Description
    PLACE_ID NUMBER(19) PK, NOT NULL 장소 고유 ID
    MEMBER_ID NUMBER(19) FK, NOT NULL 장소 추가자 ID
    PLAN_ID NUMBER(19) FK, NOT NULL 소속된 계획 ID
    NAME VARCHAR2(255) 장소 이름
    LATITUDE DECIMAL(9,6) NOT NULL 위도
    LONGITUDE DECIMAL(9,6) NOT NULL 경도
    TRAVEL_DATE DATE NOT NULL 방문 예정일
    ORDER_IN_DAY NUMBER(10) NOT NULL 일자 내 방문 순서
